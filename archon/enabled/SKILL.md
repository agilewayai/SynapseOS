# ⚡ Archon 使能层 (Enabler Layer)

> **版本**: 1.3.0 | **基于**: 认知内核层 v4.5.0 (27 模型) | **设计**: 仿生脑(026) + 分层(001)

## 定位

Archon 是采访层和 Xuan Master 认知内核层之间的**执行力外壳**。它接收经过采访校准的精准问题，选择最合适的模型管道，驱动分析执行、动作调用与结构化产出。

```
Archon 校准入口        →  发现问题 + 校准问题 + 收集全维度信息
  ↓ Interview Summary
Archon 执行引擎        →  模型选择 + 管道执行 + 动作调用 + 合成产出
  ↓
Xuan Master 核心       →  27 个深层认知模型
```

## 设计哲学 — Unix 原则驱动的执行引擎

使能层的设计深受 **Unix 设计哲学**的启发。10条经过 50 年检验的系统原则与 Xuan Master 内核模型形成了深层映射——本质上，两者都是关于 **如何应对复杂性** 的方法论：

| Unix 原则 | 使能层体现 | 内核模型支撑 |
|----------|-----------|-------------|
| **模块原则** | 每个模型独立加载、独立产出、独立可用 | 001分层 · 010奥卡姆 |
| **组合原则** | 管道模式 — 模型 A 的输出 → 模型 B 的输入 | 002流动 · 021SECI |
| **统一接口** | 标准化模型调用: context + depth → finding + principle_map | 003状态机 · 001分层 |
| **清晰原则** | 可见推理 — 每个结论标注「来自哪个模型的哪条原则」 | 006第一性原理 · 016双系统 |
| **分离原则** | Phase 1 显式分离 Policy(用户偏好) / Mechanism(引擎选择) | 001分层 |
| **简洁原则** | 轻量外壳 — 不增加新认知内容，只增加行动编排 | 010奥卡姆 · 009最小阻力 |
| **透明原则** | 每个 Phase 的中间状态对外可见——可审查、可调试 | 011反馈循环 · 003状态机 |
| **健壮原则** | 三级降级策略 — 模型不可用时替换→降深度→诚实告知 | 025高可用 · 008熵增 |
| **表示原则** | 学习记录 — 每次运行的结果反馈到「问题→模型」映射表中 | (数据驱动进化) |
| **最小意外** | 模板标准化 — 同类问题每次产出的格式、深度、结构一致 | 009最小阻力 |

## 执行协议 (Agent 加载本 Skill 后自动遵循)

### Phase 0: 采访触发判断（新增）

在进入 Phase 1 之前，先判断是否需要启动采访层：

**自动触发采访层的条件**（满足任一即触发）：
- 问题陈述 < 30 字且未包含具体约束
- 问题中有模糊词（"搞好""想办法""不知道怎么办"）
- 涉及多人但未说明谁的意见优先
- 有明显的情绪信号但原因不明

**如果触发**：加载 `[`interview`](../interview/SKILL.md)`，执行三阶段采访，产出 Interview Summary，然后基于 Summary 继续 Phase 1。

**如果不触发**：直接进入 Phase 1。

- 合成策略（互补视角/递进深化/对立平衡/递归套用/管道执行）
- 降级备选方案（见「降级策略」section）

向用户展示诊断结果（Policy + Mechanism 的合并摘要），征得确认后进入 Phase 2。

### Phase 2: 模型逐个执行

**目标**：对每个选中的模型，按框架展开分析。根据 Phase 1 选择的执行模式，有两种执行路径。

**标准化模型接口**（一切皆文件原则）：

无论哪种模式，每个模型都遵循统一的调用契约：

```
输入 (Input):    context = 被分析的问题/场景/上游模型的精炼输出
                 depth   = quick / standard / deep
输出 (Output):   finding       = 核心发现（一句话摘要）
                 principle_map = 发现→模型原则 的映射表
                 score         = 该模型对本问题的适配度自评 (0-10)
                 next          = 推荐的下游模型（管道模式用）
```

**模式 A：并行执行 (Parallel — 默认)**

模型各自独立分析同一个原始问题，互不依赖。

步骤：
1. 加载第一个模型：`[`NNN-...`](../NNN-.../SKILL.md)`
2. 按模型结构展开：
   - **核心定义** → 框定分析边界
   - **核心原则** → 作为分析检查项逐条对照
   - **实践要点** → 验证分析的实用性
   - **典型案例** → 做类比推理
3. 按标准接口格式产出该模型的独立分析段落
4. 对栈中每个模型重复（可并行思考，逐个输出）
5. 全部产出后进入 Phase 3 合成

**模式 B：管道执行 (Pipeline)**

模型串行依赖——每个模型接收上游模型的精炼输出，而非原始问题。

步骤：
1. 加载管道第一个模型，接收「原始问题」作为 context
2. 执行模型分析，产出标准接口格式的输出
3. 将输出的 `finding` + `principle_map` + `next` 打包为精炼上下文
4. 用精炼上下文替换原始 context，传给管道下一个模型
5. 重复直到管道结束
6. 管道的最终输出 = Phase 3 的合成输入

> **管道模式示例**：
> 问题: "系统性能下降" → 
> [BFS-DFS(017)] 广扫所有可能原因 → context="CPU、内存、IO 三个方向，CPU 最可疑" →
> [帕累托(004)] 聚焦 CPU → context="20% 的查询消耗 80% CPU，与缓存配置相关" →
> [流动(002)] 分析缓存流 → context="缓存穿透导致 DB 层流量是预期的 4 倍" →
> 最终输出: 一个递进的诊断链，每一步都比上一步更精确

**规则**：
- 逐个加载模型，每个分析完再加载下一个
- 每个模型的分析必须显式标注使用了哪个原则/案例
- 管道模式下，下游模型的输入是上游的「精炼输出」，不要重复分析原始问题

**备选模型加载**：
如果主要模型不可用（无此编号的 skill），从 Phase 1b 预置的降级备选方案中选下一个。如果连续 2 个不可用，触发「降级策略」。

### Phase 3: 多模型合成

**目标**：合并多模型视角，产生 1+1>2 的洞察。

**步骤**：
1. 列出所有模型分析的关键发现
2. 选择一个合成策略（见下方）
3. 找出互补、对立、递进关系
4. 写出综合洞察——标注每个洞察来自哪些模型的交叉点

**合成策略**：

| 策略 | 何时使用 | 方法 |
|------|---------|------|
| **互补视角** | 多模型从不同维度分析同一问题 | 拼成全景图——合并不重叠的发现 |
| **递进深化** | 从宏观到微观 | 模型A粗筛 → 模型B聚焦 → 模型C细查 |
| **对立平衡** | 两个模型得出矛盾结论 | 揭示 trade-off——为什么会有矛盾 |
| **递归套用** | 一个模型审视另一个 | 元认知(019)观察 SECI(021) → 找到转化瓶颈 |

**参考文件**：`[`enabled`](SKILL.md) → `references/synthesis-strategies.md``

### Phase 4: 结构化产出

**目标**：将合成洞察转化为特定格式的交付物。

**步骤**：
1. 根据问题类型，选择一个产出模板
2. **根据 Phase 1 选择的呈现模式**，应用对应的格式规则
3. 确保产出包含：结论 + 推理过程 + 行动建议

#### 呈现模式格式规则

**`expert` 模式**（默认）：
- 每章标注驱动模型编号和名称："⚛️ 第一章（驱动模型：006 第一性原理）"
- 保留核心原则引用："流动模型的瓶颈法则告诉我们..."
- 底部署名方法论来源

**`consumer` 模式**：
- ❌ 不出现任何模型编号（006/001/012 等）
- ❌ 不出现模型名称（第一性原理/博弈论/双系统等）
- ❌ 不出现"驱动模型""核心原则""实践要点"等框架术语
- ✅ 保留章节结构和逻辑递进
- ✅ 用日常语言表达同样的逻辑——"从最根本的地方开始想""换个角度看""人有两个大脑"
- ✅ 用故事、比喻、场景替代抽象概念——"就像水流总会找到最低的通道"
- ✅ 结论可以直接朗读给目标受众听而不用解释术语

> **本质**：consumer 模式不是"写一份新的"——是用**完全不同的语言**表达**完全相同的逻辑结构**。

**内置产出模板**：

| 模板 | 适用场景 | 结构 |
|------|---------|------|
| **分析报告** | 问题诊断、深度调研 | 背景→全景→根因→多视角→综合结论 |
| **设计文档** | 系统/产品/流程设计 | 需求→架构→数据流→状态机→权衡→实施 |
| **决策框架** | 战略选择、方案评估 | 现状→因子→选项矩阵→推演→建议 |
| **复盘报告** | 项目/迭代回顾 | 发生了什么→学到了什么→下次怎么做 |
| **认知体检** | 个人/团队认知评估 | 思维模式→认知偏差→元认知→成长建议 |
| **Web Slides** | 演示分享 | Markdown→reveal.js幻灯片→公网HTTP可点击 |
| **PDF 专业版** | 存档/转发 | pandoc+weasyprint+Noto字体+封面+页码 |
| **🎞️ Web Slides** | 演示/分享/教学 | reveal.js 幻灯片 → HTTP 服务器 → Cloudflare Tunnel 公网 URL |
| **Web Slides** | 演示/分享/公网分发 | reveal.js 幻灯片 + HTTP 服务器，可直接点击访问 |

> 产出多格式交付时，飞书文档末尾追加资源链接（PDF + Web Slides URL）。详见 `feishu-doc-writer` skill。

**参考文件**：`[`enabled`](SKILL.md) → `references/output-templates.md``

```
user: "我们团队开发效率很低，帮我分析一下"

```

## 降级策略 (Degradation — 健壮原则)

源于 Unix "Robustness Through Simplicity"原则。当模型加载或执行出现异常时，按以下三级协议降级：

| 级别 | 触发条件 | 降级动作 | 用户体验 |
|:----:|---------|---------|---------|
| **L1: 替换** | 单个模型不可用（无此 skill 或加载失败） | 从 Phase 1b 预置的降级备选栈中选下一个模型 | 无感知——只是用了一个替代模型 |
| **L2: 降深度** | 连续 2 个模型不可用，或 L1 替代也失败 | 切换到 `quick` 模式——只加载 1 个模型，跳过 Phase 3 合成 | "由于部分模型不可用，已切换为快速诊断模式" |
| **L3: 诚实降级** | L2 也无法完成，或用户问题超出了所有可用模型的覆盖范围 | 明确告知用户当前可分析的范围，请求更多信息或缩小问题范围 | "这个问题超出了我当前可用的模型范围。能否聚焦到[可分析的方面]？或者提供更多关于[缺失维度]的信息？" |

> **原则**: 宁可诚实地说"我只能做到这个程度"，也不要假装分析了全部维度。
