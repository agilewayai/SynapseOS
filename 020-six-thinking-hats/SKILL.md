# 🎩 六顶思考帽（Six Thinking Hats）— 模型 020

> **版本**: 1.0.0 | **作者**: Arthur

## 核心定义

将辩论/讨论/决策中的六种基本思维模式分离为六顶"帽子"，每次只戴一顶、只从一个角度思考。核心洞察：**大脑最擅长单线程运作，而群体讨论最忌"所有人同时说不同话"**。

## 六帽体系

### 🎩 白帽（White Hat）— 信息中立
关注：事实、数据、客观信息。
- "我们知道什么？"
- "我们不知道什么？"
- "我们需要知道什么？"
- 只有事实，没有判断、没有建议、没有情感
- 双方争议时用白帽：先确立共同认知的数据基础

### 🔴 红帽（Red Hat）— 直觉与情感
关注：感觉、直觉、情绪反应。
- "我对这个方案的感觉是什么？"
- "我的直觉告诉我什么？"
- 不需要解释、不需要辩护、不需要找理由
- **这是唯一允许非理性的帽子**
- 价值：在理性之前先承认情感的存在，避免它在争论中隐秘地释放

### ⚫ 黑帽（Black Hat）— 谨慎与风险
关注：风险、问题、为什么不行。
- "这个方案有什么风险？"
- "哪里可能出错？"
- "为什么不应该这么做？"
- 天然是"批判性思维"的核心输出
- 误区：黑帽不是攻击人，是攻击方案的风险

### 🟡 黄帽（Yellow Hat）— 价值与乐观
关注：好处、价值、为什么行。
- "这个方案的好处是什么？"
- "机会在哪里？"
- "如果成功，最佳结果是什么？"
- 有建设性的乐观（不是盲目的乐观）
- 背后的逻辑：每一个方案都一定有某种价值——戴黄帽的人有义务找到它

### 🟢 绿帽（Green Hat）— 创造与可能
关注：创意、替代方案、新想法。
- "除了这个还有什么可能？"
- "如果完全重新开始呢？"
- "怎样可以既保留优点又规避风险？"
- 鱼不在池里没关系——先去想鱼还能长什么样
- 绿帽阶段不否定："是的，而且……"模式

### 🔵 蓝帽（Blue Hat）— 过程与控制
关注：流程、组织、概览。
- "我们下一步做什么？"
- "现在戴什么帽子？"
- "目标是什么？我们达到目标了吗？"
- **蓝帽是元认知在群体讨论中的具体化**
- 会议中最需要但最容易被忽略的帽子

## 覆盖领域（9 个）

| # | 领域 | 六帽价值 |
|---|------|---------|
| 1 | **会议沟通** | **核心应用场景**——结构化讨论、减少冲突 |
| 2 | **产品决策** | 全面评估方案，避免"拍脑袋"或"毁灭性争议" |
| 3 | **团队管理** | 培养多元视角、降低群体思维 |
| 4 | **战略规划** | 在乐观与谨慎之间找到平衡 |
| 5 | **教学/引导** | 训练学生多角度思考；引导复杂讨论 |
| 6 | **冲突调解** | 给对立双方一个结构化的对话框架 |
| 7 | **创意与创新** | 绿帽专门为创新留出空间 |
| 8 | **个人决策** | 自己给自己"换帽子"——多角度审视 |
| 9 | **写作/内容创作** | 不同段落/章节用不同的帽子视角展开 |

## 核心原则（6 条）

### 1. 一次一顶 → 并行思维
六帽的核心原则：**所有人都戴同一顶帽子**。不是争论"你说的是对的但你没看风险"，而是"现在我们在白帽阶段，先别谈风险"。

### 2. 不争吵只互补
六帽思想不是取悦所有人，而是**结构性地互补**。每个帽子都是思考的一个必要层面——没有哪顶帽子是"正确答案"。

### 3. 帽子是方向不是身份
"你平时就是黑帽思维者"是对六帽最大的误用。帽子是"戴上"而不是"天生"的。关键是让人戴上不习惯的帽子——强制自己看到平时忽略的角度。

### 4. 蓝帽定序
蓝帽（通常是主持人/引导者）决定什么时候戴什么帽。序列可以根据话题、会议长度、团队动态自由设计。

### 5. 帽子可以穿戴多次
不需要一定用完所有帽子，也不需要顺序固定。关键场景：回到某顶帽子再做一次（如黑帽后发现绿帽方案，黑帽再检查一次新方案）。

### 6. 时间限制
每顶帽子应该有时限（5-15 分钟）。限制确保高效——"黑帽时间只有 5 分钟，请挑最重要的三点风险"。

## 实践要点

### 经典序列（5 种）

| 场景 | 首选序列 | 时长 |
|------|---------|------|
| **快速评估一个想法** | 🟡黄→⚫黑→🟢绿→🔵蓝 | 15-20 分钟 |
| **探索新领域** | 🎩白→🟢绿→🟡黄→⚫黑→🔵蓝 | 30-40 分钟 |
| **完整提案评估** | 蓝①→🎩白→🟡黄→⚫黑→🟢绿→🔴红→🔵蓝② | 45-60 分钟 |
| **创新/头脑风暴** | 🔵蓝①→🔴红→🎩白→🟢绿→🟡黄→⚫黑→🔵蓝② | 40-50 分钟 |
| **个人决策** | 🎩白→🔴红→🟡黄→⚫黑→🟢绿→🎩白→🔵蓝 | 20-30 分钟 |

### ⚠️ 反模式

| 反模式 | 表现 | 解药 |
|--------|------|------|
| 帽子固定为人 | "他就是黑帽人格"→别人不敢说风险 | 帽子是方向不是身份，强制每个人在不同环节戴不同帽 |
| 红帽被忽略或否决 | "说出你的感觉"→"那不是事实"→否定 | 红帽发言不需要理由；听完再换白帽验证 |
| 黑帽统治一切 | 讨论永远是"这不行为什么不行" | 强制规定必须先戴黄帽再戴黑帽 |
| 蓝帽缺失 | 没有主持人定序→所有人同时乱戴帽子 | 指定一个人全程或分段戴蓝帽 |

## 典型案例

### 1. 特斯拉电池战略会议
背景：特斯拉在某周的团队讨论是否要在下一代车型中改用新电池技术。反对声和赞成声在同一个讨论中同时爆炸——这是一个经典的六帽场景。
用了六帽：
- 🎩白（10 分钟）：新电池的能量密度、成本、供应链能力——事实基础
- 🟡黄（10 分钟）：更长的续航、更低的成本、差异化优势
- ⚫黑（10 分钟）：技术成熟度不够、供应商只有一家、换电池导致重新认证车壳
- 🟢绿（10 分钟）：能不能分阶段切换？高端先用？继续探索第三条路径？
- 🔴红（5 分钟）：每人只说一句"我的直觉"。其中一人说："我直觉觉得这个技术在 3 年内成熟，但我们不需要第一波。"——这句话缩短了整个决策时间。
- 🔵蓝（5 分钟）：结论→启动并行研究，但不在下一代产品中锁定设计

### 2. 启动新产品的团队讨论
背景：产品团队讨论是否要推出一款针对"年轻人"的新功能——但团队里老人和新人的观点完全不同。
- 第一阶段混战：老人说"做这个没用"，新人说"你不懂年轻人"——没有信息交换，只有身份立场。
- 第二阶段干预：引导者用六帽介入。
  - 🎩白：放出了用户调研数据：该功能在 18-24 岁用户里使用率 73%，但留存率仅 12%。
  - ⚫黑：老人列出了"这部分用户付费率低"的风险。
  - 🟡黄：新人说"如果做成功了，可以撬动 A 世代大市场"。
  - 🟢绿：团队最终提出"免费版本可解锁该功能 + 订阅用户优先体验"的方案。
结果：双方都觉得自己被听到了功能重新定位后上线，付费转化率保持在老产品的 90%。

### 3. 冲突调解
背景：两个技术团队就"用 MySQL 还是 PostgreSQL"争论了三周。每边的工程师都列出了大量的事实和数据证明对方不行——典型的"用白帽内容做黑帽发言"。
引导者介入：
- 🔵蓝：先设规矩——"
- 🎩白：所有"我们不用 X，因为 X 慢/不稳定/不支持 Y"属于黑帽内容不在此环节发言。白帽环节只允许列事实："
  - 左边：MySQL 支持 X 事务，内存 16G
  - 右边：PostgreSQL 有 Y 扩展，我们的数据规模 100 万
- ⚫黑：列风险——MySQL 的分片扩展成本、PostgreSQL 的维护团队经验不足
- 🟡黄：列价值——MySQL 的兼容性和现有基础设施；PostgreSQL 的长期扩展空间
结果：不再是"你的数据库好还是我的好"而是"半年内用 MySQL，一年后决定是否迁移到 PostgreSQL"。

### 4. IDEO 设计冲刺
设计公司 IDEO 的许多设计冲刺中内置了类似六帽的"单线程思维切换"：
- 第一小时：白帽（调研分享、用户数据）
- 第二小时：绿帽（疯狂头脑风暴")
- 第三小时：黄帽（选出最喜欢的想法）
- 第四小时：黑帽（测试方案在现实中会怎样失败）
- 最后：红帽（团队直觉投票）
IDEO 没有明确说自己在用六帽，但本质上完全复制了它的结构。说明六帽是一种"被独立发现的普遍模式"。

### 5. 技术选型评估
背景：CTO 评估是否要从单体架构拆成微服务。
- 🎩白：当前系统指标、微服务团队规模、迁移时间预估
- 🟡黄：团队能力增长、独立部署、技术吸引力
- ⚫黑：迁移期间稳定性风险、分布式复杂度、运维成本
- 🟢绿：也许不需要完全拆？小部分功能先迁移？或者 Serverless？
- 🔴红：CTO"我直觉再等一年"
结果：推迟迁移计划，但在那之前启动"微服务知识计划"——六帽决策后团队无人有异议，因为每一面都被听过了。

### 6. 个人买房决策
背景：一个人纠结买一套价格偏高通勤少 vs 价格合理通勤多 的房子。
用**个人版六帽**排序：
- 🎩白：两套房的数据信息
- 🔴红：先承认自己的情感——想要好的生活、不希望太大压力
- 🟡黄：高价低通勤房的价值——时间、享受、周围伙伴
- ⚫黑：高价房的风险——每月还款压力、万一失业
- 🟢绿：有没有其他选择——远程工作三天、分期装修、先租后买
- 🔵蓝：总结——先看低价通勤房，除非收入结构有确定性的变化

### 7. 年度述职复盘
用六帽做年度回顾：
- 🎩白：数据——收入、绩效、项目数、学习时长
- 🔴红：最开心的项目；最沮丧的失败
- 🟡黄：今年做得最对的决策
- ⚫黑：今年做得最后悔的决策
- 🟢绿：明年有哪些自己之前没试过的方向
- 🔵蓝：总结——明年最大的两件事

## 跨模型关联

### 推荐场景组合
> **产品价值评估(含风险)**: 020 + 015 + 004 + 014
> **高效会议/群体讨论**: 020 + 019 + 012 + 014
> **团队决策/投资评估**: 020 + 018 + 005 + 016
> **结构化表达/提案**: 020 + 022 + 014 + 005

- **六顶帽(020) + 元认知(019)**：蓝帽完全就是元认知在群体讨论中的**具体化**。掌握元认知后，你在会议中自动知道现在是否"需要换帽"。
- **六顶帽(020) + 双系统(016)**：红帽给系统 1（直觉/情感）一个合法的发言窗口；黑帽+黄帽让系统 2 有结构地输出。红帽+白帽的组合有效抑制"直觉伪装成事实"的偏见。
- **六顶帽(020) + 逆向(005)**：黑帽时段专门用逆向思维找"如果我们错了会怎样"。黄帽时段也可以逆向："如果我们做成了，最耀眼的成果是什么？"
- **六顶帽(020) + 5W2H(014)**：白帽时段用 5W2H 结构化收集信息。蓝帽定序也可以参考 5W2H（何时何地何人）。
- **六顶帽(020) + SWOT(018)**：SWOT 是静态的四象限诊断，六帽是动态的过程。组合：白帽(找 S+W) → 黄帽(找 O) → 黑帽(找 T) → 绿帽(创造策略) → 蓝帽(总结)。
- **六顶帽(020) + 博弈论(012)**：当团队"为什么总争论同一点"，往往是立场导致的信息不对称问题。六帽通过切换帽子，切断立场与观点的绑定——这是博弈论中的"结构化解冲突"。

## 💡 实战练习

### 初级
初级：下周的会议中，尝试"六顶帽"结构——给每个议题分配一顶帽。注意：不是每人戴一顶，是所有人在同一时间戴同一帽。体验：这种方式和"自由讨论"的效率差异。

### 中级
用六顶帽审视你最近的一个重要决定。分别从白(事实)、红(直觉)、黑(风险)、黄(机会)、绿(创意)、蓝(过程)六个角度重新看——每个角度能发现什么之前没看到的？

### 高级
为团队设计一个"帽子轮转"议事规则。关键：第一次可能很尴尬——坚持 3 次，观察团队讨论的质量是否提升。

### 反思
六顶帽的敌人是"所有人都假装在戴帽但其实还在用自己的惯性思维"。你怎么判断一个人是"戴着黑帽"还是"本身就是悲观人格"？
