# 多模型合成策略 (Multi-Model Synthesis Strategies)

> 使能层 Phase 3 的参考文件 — 如何将多模型分析转化为全景洞察

---

## 合成策略总览

| 策略 | 适合场景 | 核心动作 | 产出形态 |
|------|---------|---------|---------|
| **互补视角** | 多模型从不同维度分析同一对象 | 拼合 → 找覆盖盲区 → 补充 | 全景地图 |
| **递进深化** | 从粗到细、从表到里 | 粗筛 → 聚焦 → 细查 | 诊断链条 |
| **对立平衡** | 两个模型得出了矛盾结论 | 辨析 tension → 找到和解点 | Trade-off 矩阵 |
| **递归套用** | 一个模型审视另一个模型的应用过程 | 元视角交叉 | 二阶洞察 |

---

## 策略一：互补视角 (Complementary Perspectives)

### 何时使用
多个模型从不同的维度（结构/动态/认知/策略）分析同一个问题，维度之间不重叠。

### 方法
1. 列出每个模型关注的维度
2. 拼成全景图：标注每个洞察的来源模型和维度
3. 找覆盖盲区："还有哪个维度没有被任何模型覆盖？"
4. 如果盲区重要，建议补充模型

### 示例：分析一个软件系统的脆弱性

**模型栈**: 001(分层) + 002(流动) + 008(熵增) + 025(高可用)

| 模型 | 维度 | 发现 |
|------|------|------|
| 001 分层 | 静态结构 | 数据库层和应用层共享相同的故障域 |
| 002 流动 | 动态行为 | 流量峰值导致队列积压，但只在周二下午 3 点 |
| 008 熵增 | 退化趋势 | 日志文件 6 个月增长 4 倍，无自动清理机制 |
| 025 高可用 | 韧性设计 | 当前无自动故障转移——手工切换需要 15 分钟 |

**合成**：
```
全景洞察：系统的脆弱性存在于三个层次——
- 结构层：共享故障域 → 风险放大
- 动态层：不可预测的峰值 → 反应被动
- 趋势层：缓慢退化的基础设施 → 潜在崩溃

三个维度互相独立但相互放大：
共享故障域(001) + 无自动切换(025) = 单点故障杀伤力被最大化
峰值积压(002) + 日志膨胀(008) = 峰值时的恢复窗口更长
```

### 核验问题
- 每个模型的发现是否指向不同的维度？（如果两个模型说的是同一件事，换互补策略的用法不对）
- 全景图中是否有明显空白？（比如没有认知维度的分析）
- 各发现之间是否有"放大效应"——一个维度的问题让另一个维度更严重？

## 策略二：递进深化 (Progressive Deepening)

### 何时使用
问题需要从宏观到微观、从模糊到精确的逐步收缩过程。

### 方法
1. 模型A粗筛：确定问题的大致范围（"问题出在哪个系统"）
2. 模型B聚焦：缩小范围（"是系统的哪个部分"）
3. 模型C细查：精准定位（"根因是什么"）
4. 每一步基于上一步的输出——形成诊断链条

### 示例：诊断项目延期

**模型栈**: BFS-DFS(017) → 帕累托(004) → 逆向(005) → 流动(002)

```
Step 1 - BFS-DFS(017) 粗筛：
  BFS 扫描所有可能延期原因 → 列出 8 个候选方向
  DFS 深钻最有希望的方向 → "需求变更"和"外部依赖"是两个最可疑的方向

Step 2 - 帕累托(004) 聚焦：
  统计过去 6 个月的延期数据 → 
  两个人在做的任务贡献了 70% 的延期风险（跨部门协调 + 外部 API 集成）
  → 排查范围从"整个团队"收缩到"两个人的两类任务"

Step 3 - 逆向(005) 反向验证：
  假设这两类任务没有延期 → 项目会提前多少？
  推算：减少延期 = 项目进度至少快 40%
  → 确认：这是一个高杠杆问题点

Step 4 - 流动(002) 根因：
  画这两个人的工作流 →
  瓶颈点：等待外部响应时间占任务总时长的 60%
  → 根因不是"他们太慢"，而是"工作模式"——串行等待
```

**合成**：
```
诊断链: BFS扫全貌 → 帕累托收缩范围 → 逆向验证杠杆 → 流动精准定位根因

关键洞察: "项目延期"表面上是一个进度管理问题
  → 实际上是工作模式问题：跨部门协调用串行模式
  → 解决方案不是"加人"或"让他们更努力"
  → 而是"改工作模式"——用异步协作 + 提前锁定外部依赖的窗口期
```

### 核验问题
- 每一步是否真正"缩小了范围"？如果第一步说"所有可能"，第二步又说"所有可能"，递进没有发生
- 最后一步是否找到了可行动的切入点？

## 策略三：对立平衡 (Tension Resolution)

### 何时使用
两个模型从不同原则出发，得出了看起来矛盾的结论。

### 方法
1. 明确陈述两个模型的矛盾点
2. 检查：矛盾是"真实矛盾"还是"在不同前提下的不同结论"？
3. 如果是真实矛盾 → 揭示 trade-off 的本质
4. 输出：不是"哪个对"，而是"在什么条件下选哪条路"

### 示例：技术债务管理

**模型栈**: 熵增(008) + 奥卡姆(010)

```
矛盾: 
  熵增(008) 说 → "如果不持续清理技术债务，系统必然退化"
  奥卡姆(010) 说 → "如无必要勿增实体——不应该为了清理而清理"

辨析:
  这是一个"在什么条件下选哪条路"的问题——
  - 熵增说的是趋势：放着不管就会更糟（TRUE）
  - 奥卡姆说的是选择：不是所有的"债务"都值得立即还（ALSO TRUE）

解决:
  两条原则在不同条件下适用：
  ┌──────────────┬─────────────────────────┬─────────────────────┐
  │ 高影响 + 高频 │  熵增主导 → 立即清理     │ "每次部署都碰到的bug" │
  │ 高影响 + 低频 │  平衡点 → 列入最近清理    │ "季度结算时的性能问题" │
  │ 低影响 + 高频 │  平衡点 → 自动化处理      │ "format不一致"       │
  │ 低影响 + 低频 │  奥卡姆主导 → 不处理      │ "从未触发过的死代码" │
  └──────────────┴─────────────────────────┴─────────────────────┘
```

**合成**：
```
Trade-off 的本质：不是"清理 vs 不清理"，而是"何时清理、清理到什么程度"
→ 熵增提供方向（必须持续反熵）
→ 奥卡姆提供克制（只反必要的熵）
```

### 关键技巧
- 如果两个模型给出了矛盾结论，先检查它们的前提条件是否一致
- 创建一个 2x2 矩阵来可视化 trade-off
- 输出要让用户看到"选择的代价"而不是"正确的答案"

## 策略四：递归套用 (Recursive Application)

### 何时使用
用一个元级模型来审视另一个模型的**应用过程**本身——不只是分析问题，而是分析"分析方式"。

### 方法
1. 用模型A分析问题（得到直接洞察）
2. 用模型B审视步骤 1 的过程（得到二阶洞察）
3. 二阶洞察通常比直接洞察更有价值——它揭示的是"思维模式"而非"问题本身"

### 示例：用元认知审视 SECI 知识转化

**模型栈**: SECI(021) → 元认知(019)

```
Step 1 - SECI(021) 分析知识管理流程:
  - 共同化阶段：信息流动顺畅，团队有良好的讨论文化
  - 表出化阶段：文档产出不稳定——取决于"谁有空写"
  - 联结化阶段：知识碎片分散在多个工具中，无统一索引
  - 内隐化阶段：新成员需要 3 个月才能独立上手

Step 2 - 元认知(019) 审视这个分析本身:
  - "这个 SECI 分析本身，我写成了文档吗？" → 没有。我刚刚在脑子里分析完就输出结论了
  - "这说明什么？" → 我自己就在做 SECI 循环断裂的示范——
    共同化了（头脑中分析）但跳过了表出化（没有先写文档再输出）
  - "如果我把这个 SECI 分析本身先写成一页纸再输出，结果会有什么不同？"
    → 可能在写的过程中发现自己遗漏了"为什么文档产出不稳定"的更深层原因
```

**合成**：
```
直接洞察: SECI 循环在表出化阶段断裂
元洞察:   "发现这个问题"的方式本身也是 SECI 断裂的表现
          → 如果我用 SECI 的纪律来分析 SECI 的问题，
             第一步应该先把初始分析写下来而不是只说结论

行动建议: 不仅仅是修复团队的 SECI 流程，还要在每一次"用模型分析问题"时，
         先把分析本身做表出化（写下来），建立"元认知的表出化习惯"
```

### 何时使用递归套用
- 问题涉及"思维方式的改进"时（因为套用会揭示思考盲区）
- 019(元认知)是最适合作为套用器的模型——它天生就是"观察观察者"的
- 026(仿生脑)的反思维也可以作为套用器

## 合成质量检查清单

在输出 Phase 4 的结构化产物之前，用这个清单检查合成质量：

- [ ] 每个模型的发现是否都在合成中有所体现？（没有遗漏）
- [ ] 合成是否超越了"把四个发现放在一起"的层面？（有真正的交叉 insight）
- [ ] 是否找到了模型之间的放大效应或矛盾？如果完全没有，可能选错了合成策略
- [ ] 合成结论是否可操作？（用户看完知道"下一步做什么"）
- [ ] 合成过程是否可见？（用户能追溯每个洞察来自哪个模型）
