## 关键洞察一：组合原则 → 「模型管道」(Model Pipeline)

> **Unix 启发**: `cat file | grep error | wc -l` — 简单工具通过管道组合成强大工作流

### 使能层当前状态
Phase 2 逐个加载模型 → Phase 3 合并视角。模型之间是**并行关系**（各自独立分析同一问题），然后手动合成。

### 缺少的能力：模型管道
模型之间可以有**串行关系**——一个模型的输出作为下一个模型的输入，像 Unix 管道一样：

```
用户问题
  → [001 分层] 分解为子问题
    → [002 流动] 分析每个子问题的瓶颈
      → [004 帕累托] 识别 20% 关键瓶颈
        → [015 精益] 设计消除方案
          → 结构化产出
```

这和当前的"并行合成"是完全不同的执行模式：
- 并行合成: 每个模型看到的是"原始问题"
- 管道执行: 每个模型看到的是"上游模型精炼后的输出"

### 具体改进：在 SKILL.md 中增加第六种执行模式

在现有的「互补视角/递进深化/对立平衡/递归套用」四种合成策略之外，增加：

| 策略 | 何时使用 | 模型关系 | 产出形态 |
|------|---------|---------|---------|
| **管道执行 (Pipeline)** | 问题可以分解为递进步骤，每步需要不同模型 | 串行依赖 A→B→C→D | 递进诊断链 |
| **并行合成 (Synthesis)** | 多模型从不同维度看同一问题 | 并行独立 A∥B∥C | 全景图 |

### 管道模式的执行协议

```
Phase 2-bis: 管道模式
  对管道中的每个模型:
    1. 加载模型
    2. 接收上游模型的"精炼输出"作为输入（而非原始问题）
    3. 用模型框架分析精炼输出
    4. 产出的不是"完整分析"而是"聚焦分析"
    5. 将聚焦分析传给下游
```

## 关键洞察二：一切皆文件 → 统一的模型调用接口

> **Unix 启发**: 文件、目录、设备、进程间通信都抽象为文件。统一的 `open/read/write/close` 接口。

### 使能层当前状态
每个模型通过 `skill_view(name='00-entry-NNN-...')` 手动加载，模型的"输入"和"输出"没有标准化接口。

### 具体改进：模型调用包装器

设计一个心理模型——把每个 Xuan Master 模型想象成一个 **Unix 风格的过滤器**：

```
                    ┌─────────────────┐
   问题/上下文 ───→ │  Xuan Master     │ ───→ 结构化的模型视角
                    │  模型 (过滤器)    │
                    └─────────────────┘
```

统一的"输入接口"：每个模型接收两种输入
- `context`：被分析的问题/场景/数据
- `depth`：分析深度（quick / standard / deep）

统一的"输出接口"：每个模型输出
- `finding`：核心发现（一句话摘要）
- `principle_map`：关键发现与模型原则的映射表
- `score`：这个模型对该问题的"适配度"自评（0-10）
- `next`：建议的下游模型（管道模式用）

### 对 SKILL.md 的修改

在执行协议中增加这个标准化接口的说明——让 Agent 在 Phase 2 时自动按此格式组织每个模型的输出。

## 关键洞察三：表示原则 → 数据驱动的模型选择

> **Unix 启发**: 将知识逻辑融入数据，而非程序代码。通过配置文件定义行为，而非硬编码。

### 发现：Xuan Master 原生模型中没有直接对应

27 个模型覆盖了思辨/方法/系统，但没有一个**将数据本身作为知识载体的模型**。这可能是未来的第 28 个模型方向——但先解决更紧迫的问题。

### 使能层当前状态
`model-selector.py` 的模型选择基于**静态的关键词匹配表**（硬编码的 `KEYWORD_MAP`）。

### 具体改进：从静态映射到数据学习

```
[当前] 关键词 → 硬编码的场景表 → 模型栈
[Listen] 用户反馈信号 → 模型栈 → 记录结果
[Learn]  历史结果数据 → 调整场景表的权重 → 更精准的模型栈
```

**在 `problem-diagnosis-guide.md` 末尾增加「学习记录」section**：

```markdown
## 学习记录 (持续更新)

| 场景 | 原始推荐 | 实际使用 | 用户反馈 | 调优方向 |
|------|---------|---------|---------|---------|
| 团队士气+延期 | 021+016+020+019 | 021+016+020+019 | ✓ | — |
| ... | ... | ... | ... | ... |
```

每次 Phase 5 反馈后，Agent 可以用 `patch` 更新这条记录。积累 20+ 条后，模型选择的精准度会从"专家系统"进化为"经验系统"。

## 关键洞察四：健壮原则 → 降级策略

> **Unix 启发**: 系统应能在异常情况下保持稳定。通过简洁设计和透明状态管理，使错误易于发现和修复。

### 使能层当前状态
如果某个模型的 `skill_view` 加载失败、或模型内容不适合当前问题、或合成阶段发现矛盾无法调和——**没有降级策略**。

### 具体改进：三级降级协议

在 SKILL.md 的执行协议中增加：

```
Phase 2 模型执行失败时的降级策略:
  ┌─────────────────────────────────────────────────────────┐
  │ Level 1: 替换                                           │
  │   当前模型不可用 → 从「替代模型栈」中选下一个             │
  │   （替代栈由入口路由的 52 场景中同一类别的其他组合提供）    │
  │                                                         │
  │ Level 2: 降深度                                         │
  │   所有替代模型也不可用 → 切换到「快速诊断」模式           │
  │   （只加载 1 个模型，跳过 Phase 3 合成）                  │
  │                                                         │
  │ Level 3: 诚实降级                                       │
  │   快速诊断也无法完成 → 告知用户当前可分析范围             │
  │   → 请求用户提供更多信息或缩小问题范围                    │
  └─────────────────────────────────────────────────────────┘
```

## 关键洞察五：分离原则 → Policy/Mechanism 显式分离

> **Unix 启发**: 将策略（用户需求）与机制（实现细节）分离。通过配置文件或命令行参数控制行为，而非硬编码。

### 使能层当前状态
Phase 1~5 的执行协议中，Policy 和 Mechanism 是混在一起的——比如"选择合成策略"既是用户偏好（Policy）又是执行选择（Mechanism）。

### 具体改进：在 Phase 1 中显式分离

```
Phase 1-a: 策略确认 (Policy — 用户决定)
  ├── 问题类型: [系统设计 / 决策 / 团队 / 个人 / ...]
  ├── 分析深度: [quick / standard / deep]
  ├── 偏好模型: [用户指定的模型，如果有]
  ├── 产出模板: [分析报告 / 设计文档 / 决策框架 / 复盘 / 认知体检]
  └── 执行模式: [并行合成 / 管道执行]

Phase 1-b: 机制选择 (Mechanism — Agent 自动)
  ├── 根据 Policy 匹配最优模型栈
  ├── 选择合成策略
  └── 设置降级策略
```

这让用户在 Phase 1 结束时可以**只说一个偏好词就改变整个执行方式**——比如"deep"、"pipeline"、"design-doc"。

## 核心洞察

Unix 哲学和 Xuan Master 的深层共鸣不是巧合——它们本质上都是关于**如何应对复杂性**的方法论。Unix 是在软件层面应对，Xuan Master 是在认知层面应对。使能层作为两者的桥梁，从 Unix 哲学中汲取的具体实践可以让它从一个"好的设计"变成一个"优雅的系统"。

最关键的三个借鉴：
1. **管道模式** — 从并行合成为主变为串并混合，解锁指数级更多分析路径
2. **Policy/Mechanism 分离** — 让用户在 Phase 1 结束时用一句话控制全部后续行为
3. **数据驱动** — 从静态映射进化为经验学习，让使能层用一次聪明一次
