# 🔄 状态机模型（模型 003）
> **分类**: ⚙️ 系统

> **版本**: 2.0.0（P0 扩容版） | **关键词**: 有限状态 ⋅ 事件触发 ⋅ 非法转移 ⋅ 状态爆炸 ⋅ 生命周期

## 核心定义

任何系统或实体的行为可以被建模为一系列**有限的状态（State）**，以及在这些状态之间由**事件（Event）**触发的**转移（Transition）**。系统的行为由当前状态和触发事件的组合唯一决定。

**根本洞察**：状态机最有价值的不是「允许做什么」，而是「不允许做什么」。当你明确定义了所有合法转移，你就同时发现了所有非法转移——而非法转移正是系统出 bug 的根源。大多数系统故障不是因为代码逻辑写错了，而是因为系统进入了「没有人预料到」的状态。

## 核心原则（7 条）

### 1. 状态与事件分离
状态是实体的静态属性，事件是引发变化的外部驱动。状态机模型强制区分二者，避免将「如何变化」和「当前是什么」混为一谈。

### 2. 有限可枚举
系统的所有状态在建模时应该是有限且可穷举的。如果状态数量无限或模糊，要么模型选错了，要么系统复杂度超出状态机适用范围。

### 3. 确定性 vs 非确定性
确定性有限自动机（DFA）：给定当前状态+事件，转移结果唯一确定。非确定性（NFA）：同一输入可能转移到多个状态。实际系统设计中优先使用确定性模型，降低心智负担。

### 4. 状态覆盖率
必须覆盖所有合法状态 + 非法/异常状态（error state / dead state）。否则生产环境会遇到「走到一个没人预料到的状态」——这是 bug 的首要来源。

### 5. 状态爆炸
当状态数量随系统规模指数增长（N 个布尔变量 → 2^N 个状态），状态机模型失效，需要改为层次状态机（HSM / Statechart）或基于事件的模型。

### 6. 状态的隐式 vs 显式
很多系统的状态是隐式的（多个字段的组合值表示一个语义状态）。应尽量显式建模——隐式状态是 Bug 的最大来源之一。

### 7. Entry / Exit / Do 动作（新增为独立原则）
进入状态时执行的动作（entry）、离开时（exit）、在状态中持续执行的（do）——UML 状态机的重要设计模式，避免在转移中重复逻辑。

## 实践要点

### ☑️ 检查清单
- [ ] 你系统的所有可能状态都列出来了吗？包括异常状态？
- [ ] 每个状态之间的非法转移是什么？有没有「从已取消到已发货」这样的路径？
- [ ] 状态是显式建模的，还是用 if-else 隐式表达的？
- [ ] 状态粒度是否合理？会不会太少（丢失细节）或太多（无法管理）？
- [ ] 是否有 entry/exit/do 动作？离开状态时是否需要执行清理？

### 🛠️ 工具方法

| 方法 | 用法 |
|------|------|
| **状态图** | 画出状态+转移的有向图，检查入度/出度是否合理 |
| **非法转移表** | 列出所有状态×事件的组合，标注合法/非法 |
| **状态爆炸检测** | 如果状态数量 = 2^N，考虑使用层次状态机 |
| **Guard 条件** | 转移不一定只由事件触发，加上条件（if）是工程关键扩展 |
| **Event Sourcing** | 不存当前状态，存所有历史事件——可以从事件推导出状态 |

### ⚠️ 反模式

| 反模式 | 表现 | 解药 |
|--------|------|------|
| **隐式状态机** | 看到 `if status == 'paid' and not refunded and ...` | 显式状态机建模 |
| **状态爆炸** | 状态数 = 2^N，N 个布尔变量 | 层次状态机（HSM）|
| **缺少 error state** | 系统走到了没人预料的状态 → 现 bug | 加入 error/dead state |
| **粒度不合适** | 「已创建-已支付-已完成」太粗，漏了中间步骤 | 回到用户旅程，找缺失状态 |

## 典型案例（6 个）

### 1. 电商订单生命周期
已创建 → 已支付 → 已发货 → 已送达 → 已完成。异常分支：已创建→已取消、已支付→退款中→已退款、已发货→退货中→已退货。每个分支都有 Guard（例如「退款中→已退款」需要支付渠道确认）。

### 2. HTTP 连接状态机
从浏览器角度：idle → resolving → connecting → sending → waiting → receiving → idle。每一步都有超时或错误处理。

### 3. Git 分支的生命周期
本地分支（可工作）→ 推送远程 → 创建 PR（open）→ Code Review → merged/closed → 合并后删除（dead）。每个状态都有允许的操作集合。

### 4. 用户注册流程
未注册 → 已填表单 → 已验证邮箱 → 已设密码 → 已激活 → 正常使用。非法转移：未注册 → 正常使用（跳过验证）。

### 5. 行为改变阶段模型（Prochaska & DiClemente）
前意向 → 意向 → 准备 → 行动 → 维持。非线性！复发（Relapse）从行动/维持回到前意向，这是状态机务实设计。

### 6. DevOps 部署流水线
待部署 → 部署中 → 金丝雀验证 → 健康检查通过 → 全量发布 → 完成。回滚：金丝雀/全量 → 回滚中 → 已回滚。每次转移都有 Guard（健康检查失败自动触发回滚）。

## 与其他 Xuan Master 模型的深层交互

### 同分类（系统）
- **状态机(003) + 流动(002)**：状态机定义「可能的状态」，流动定义「如何在状态之间转移」。没有流动的状态机是死的，没有状态机的流动是失控的。
- **状态机(003) + 分层(001)**：每层都有自己的状态机，层间通过事件通信。复杂系统 = 多个嵌套状态机按分层组织。
- **状态机(003) + 反馈循环(011)**：状态转移后的输出可以作为反馈进入入口，促使系统进入新的状态。反馈是状态机的「动力源」。
- **状态机(003) + 高可用(025)**：高可用的核心是状态检测 + 自动状态转移（健康→异常→恢复）。

### 跨分类
- **状态机(003) + BFS-DFS(017)**：状态空间搜索——DFS 深挖一条路径，BFS 广摸所有可能状态。两个探索策略适用于不同场景的状态机。
- **状态机(003) + RICE(024)**：RICE 的诊断流程本质上是一个状态机——问诊→检查→诊断→治疗→复查。每一步都在缩小状态空间。
- **状态机(003) + 帕累托(004)**：20% 的状态和转移占了 80% 的运行时间。找到「热点状态」，优化它们。

### 推荐场景组合
> **Bug 诊断**：003 + 005 + 009 + 017
> **系统架构设计**：003 + 001 + 002 + 008 + 025
> **故障排查/事故响应**：003 + 024 + 017 + 005
> **业务流程优化**：003 + 002 + 004 + 007
> **AI Agent 行为设计**：003 + 027 + 026 + 002

## 💡 实战练习

### 初级：画出你身边的状态机
选一个你日常使用的实体（一个项目、一张工单、一个申请）：
1. 列出所有可能的状态（至少 5 个）
2. 列出所有可能的合法转移（单向箭头连起来）
3. 标注非法转移（比如「已取消→进行中」）——思考「这个事故一旦发生会怎么样？」
4. 你是否漏了某些状态？比如「已暂停」「待复审」？

### 中级：代码中的隐式状态机
找一个你最近写过的代码文件（或任何项目）：
1. 搜索所有 `if status ==`、`if state ==`、`if phase ==` 的语句
2. 把所有条件分支整理出来——这就是一个隐式状态机
3. 画出显式的状态图，对比隐式的 if-else 实现
4. 有没有发现遗漏的状态分支？有没有非法转移没有处理？

### 高级：系统级状态完整性
选一个你熟悉的复杂系统（API 层、CI/CD 流水线、订单系统）：
1. 为它的核心实体绘制完整的有限状态机
2. 检查每个状态是否有 entry 动作（进入时该做什么）和 exit 动作（离开时需做什么）
3. 添加 Guard 条件到每个转移
4. 验证：如果系统从任意状态接收任意事件，结果是否可预测？

### 反思
- 你上一次 debug 的 bug 会不会是因为系统走了你没有预料到的状态？如果是，你会在哪里加一个 Guard 来阻止这个转移？
