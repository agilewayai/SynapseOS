# 🔁 模型 007：迭代思维模型 (Iterative Thinking / Evolution by Small Steps)
> **分类**: 🛠️ 方法

### 核心定义

不追求一次性得到完美答案，而是**通过重复的小循环（Plan → Do → Check → Adjust）逐步逼近目标**。每个循环都基于上一轮的结果调整方向，让系统在反馈中自修正、自演化。迭代思维的核心信念是：你不知道正确答案，但你知道下一步该试什么。

### 多元领域映射

| 领域 | 表现形式 |
|------|---------|
| **软件工程** | Agile/Scrum 的迭代冲刺（Sprint）；持续集成/持续交付（CI/CD — 每一次提交迭代一次交付）；重构（每次小改动后保持可运行状态——「童子军原则」）；DDD 的不断演进的通用语言与界限上下文 |
| **产品设计** | MVP → 用户反馈 → 迭代；精益创业的 Build-Measure-Learn 循环；Rapid Prototyping（快速原型——做出来测试→改→再测）；A/B 测试——一次迭代一个变量 |
| **管理学** | PDCA 循环（戴明环）：Plan → Do → Check → Act（最经典的迭代管理框架）；OKR 的月度/季度迭代（不是一年定一次）；Kaizen（持续改善——丰田生产系统的核心，每天小改进 vs 阶段大改革）|
| **认知科学** | 贝叶斯认知（Bayesian Updating）：信念不是静态的，而是在新证据下逐步更新——后验概率 = 先验概率 × 似然度。这是人类认知的迭代本质 |
| **生物学/进化论** | 自然选择（达尔文）：变异（小步变化）→ 选择（环境筛选）→ 遗传（保留有利变异的微小累积）→ 几十亿次迭代后产生人类。进化是最纯粹的迭代：没有设计者，没有最终目标，只有每个小循环对当前环境的适应 |
| **人工智能/ML** | 梯度下降（Gradient Descent）：沿着误差曲面的梯度方向每次微调参数，经过数千到数百万步迭代收敛到最优解；强化学习（RL）：Agent 在环境中行动→获取奖励→更新策略→再行动——每一步都是基于上一步策略的改进 |
| **数学** | 牛顿法/数值逼近：不能直接求解的方程，从一个初始猜测开始迭代逼近（x_{n+1} = x_n - f(x_n)/f'(x_n)）；蒙特卡罗方法：大量随机采样迭代逼近解析解 |
| **物理学** | 模拟退火（Simulated Annealing）：类比冶金退火过程，以一定概率接受更差的解来跳出局部最优，再逐步降低「温度」→ 收敛到全局最优——这是带「探索」机制的迭代 |
| **写作/创意** | 「烂初稿」（Vomit Draft / Shitty First Draft）方法论：先写出最烂的版本→改→再改→改到能看；海明威的名言：「所有初稿都是垃圾」——但不写出垃圾就改不出好的。迭代写作承认第一次必然差，但拒绝接受「只能这么差」|
| **创业** | Lean Startup 的「失败要快、失败要便宜」：每次尝试都是一个小迭代，快速验证或证伪假设。失败的迭代不是「失败」，是学到了「这条路不通」的信息——代价可控 |
| **医学** | 临床试验的 Phase I → II → III → IV：从小样本安全性验证开始，逐步扩大范围验证疗效和长期影响。每一步的结论决定是否进入下一步（Go/No-Go 决策）|
| **军事** | OODA 循环（Observe → Orient → Decide → Act — Boyd）：更快地完成你的 OODA 循环，就进入对手的 OODA 循环内部（get inside the opponent's OODA loop），让对手始终在反应。空战格斗中，谁的迭代频率高谁赢 |
| **教育** | 精通学习（Mastery Learning）：学生通过反复练习→反馈→纠错→再练习的迭代过程逐步掌握知识，而非一次考试定成绩。间隔重复（Spaced Repetition）的迭代本质：每次复习都基于上一次的遗忘曲线调整 |
| **工程系统设计** | DevOps 的「改善三明治」：把开发→部署→运营→反馈当作连续迭代，不是三阶段的接力棒。ChatOps 的「人+机器协作迭代」：自动化监控→告警→人工介入→自动化修复→自动化验证 |
| **天文学** | 天体观测中的迭代逼近：每次改进望远镜/算法只能获得小幅精度提升，但经过 400 年的迭代（伽利略→哈勃→韦伯→…），单次观测的精度从角分提升到亚角秒 |

### 核心原则

1. **反馈环是引擎** — 迭代的质量不取决于第一次尝试的好坏，取决于反馈环的速度和灵敏度。反馈越快，每次学习越密集，收敛越快
2. **小步快跑 > 大步流星** — 步长越大越好是反直觉的常见错误。大步子：(1) 失败代价高 (2) 难以定位问题（改了太多变量）(3) 心理阻力大。小步子：快速证伪、低代价容错、天然支持并行。利特尔法则的推论：小批量 = 低延迟 = 快反馈
3. **收敛 ≠ 到达** — 迭代可以收敛到局部最优解而不是全局最优。梯度下降在非凸曲面会停留在局部极值。需要引入「探索」（temperature / epsilon-greedy / 模拟退火）来跳出局部
4. **迭代的代价平衡** — 迭代不是免费午餐。每次迭代有切换成本（context switch）、验证成本（测试/部署/审核）、反馈成本（等待用户/数据）。当迭代成本超过迭代收益时，更少但更大的迭代更优
5. **迭代的方向性** — 迭代不是原地转圈。「做某事 100 次仍然差」和「根据反馈调整了 100 次」有本质区别。迭代需要外部信号（反馈）来提供方向
6. **增量 vs 迭代** — 增量（Incremental）是量变（每次加一点点新功能），迭代（Iterative）是质变（每次基于反馈重构之前的方案）。好的迭代过程既做增量也做迭代：保持系统可运行的同时不断重构

### 实践要点

- **最小可行迭代（MVI）**：不是「最小可用产品（MVP）」，而是每次迭代中最小可行的改进步骤。问问自己：「如果只能改一件事，我改什么？」——这是帮助收敛到关键杠杆点的思考工具
- **迭代节奏**：好的节奏不是越短越好。太短→反馈噪声大（如每小时 deploy 一次但每次都在修 typo）；太长→反馈太慢（如一年发布一次，发现方向错了已经来不及）。找到适合你系统/团队的节奏更重要
- **失败的迭代 ≠ 失败的项目**：迭代的每一次循环都可能返回「这个假设是错的」——这不是项目的失败，这是迭代的价值兑现。关键是你有没有在迭代开始时定义了「什么算失败？」（假说的可证伪性）
- **反模式：迭代变成了原地打转** — 每个 Sprint 都在修同样的 Bug，每次重构都改回原来的结构，每次优化都是同样的思路。这是「假迭代」——有循环但没有学习。添加「回顾」（Retrospective）环节，明确记录「这次学到的和下次不做的」
- **反模式：迭代没有终点** — 迭代不是永远改进的借口。在达到「足够好」后继续迭代是效率损失。好的迭代需要定义收敛条件（满足什么标准就停止？成本曲线什么时候变平？）
- **并行迭代 vs 串行迭代**：串行（A→B→C→D）：每一步依赖上一步结果，适合因果链清晰的场景（如药物研发）；并行（A∥B∥C∥D）：独立模块同时迭代，适合松耦合系统（如微服务、独立功能模块）
- **启发**：当你认为自己「在迭代」但进展缓慢时，不要问「我做得够不够快？」，而要问「我的反馈环在哪里？它够不够快？我的学习是真的是积累还是机械重复？」
- **迭代的元原则**：迭代思维本身也需要迭代——定期反思「当前的迭代方式是否适合当前的问题？」

### 典型案例

1. **PDCA 循环在丰田生产系统中的应用**：每一个工位、每一个班组每天执行 Plan（今天的目标和计划）→ Do（执行）→ Check（比对结果→观察到偏差立即停线）→ Act（标准化成功的做法/调整失败的做法）。这个连续循环使丰田在「没有革命性突破」的情况下，通过数万次小迭代成为全球最精益的制造体系
2. **深度学习模型的训练**：每一个 Epoch 就是一次迭代：Forward（当前参数输出预测）→ Loss（计算预测与真实值的差距）→ Backward（计算梯度——各参数的「改进方向」）→ Update（微调所有参数）→ 下一个 Epoch。经过 50-100 次 Epoch 迭代，模型从随机噪声收敛到可用的效果。如果不迭代，一次性训练是不可能的
3. **Git 的 commit 哲学**：好的 commit 不是「一次写了一万行然后一次性提交」，而是每次小改动、有意义的原子 commit——每个 commit 都是可运行的，每个 commit message 描述了迭代的理由。`git rebase -i` 是对迭代历史的再迭代（合并小迭代、重写 commit message）
4. **Figma 的版本历史和分支**：设计师不需要一次性完成「完美设计」再交付。每一次拖动/修改都是一个小小的迭代。Figma 的 auto-save + version history 让每个小迭代都被记录，随时可以回退到任意中间状态——这是迭代思维在工具设计中的体现
5. **儿童学习走路**：不是某一天突然会走路了。尝试站立→摔倒→尝试两小步→摔倒→尝试三步→摔倒→走两步→摔倒→走五步……每一次摔倒都是反馈，每一次尝试都是基于上次经验的小调整。从不会走到可以跑，经历了数千次迭代循环
6. **SpaceX 的 Starship 原型迭代**：从 Starhopper（单引擎、几十米跳跃）→ SN5（单引擎、150m 跳跃）→ SN8（三引擎、12.5km 高空飞行）→ SN15（全流程成功着陆）→ … 每一次迭代都基于上一次失败的教训（SN8 着陆爆炸→增压系统改进→SN9→又爆炸→Header Tank 设计改进→SN10→部分成功→连续迭代）——失败是迭代中获取反馈的昂贵形式，但它的信息密度无可替代

## 💡 实战练习

### 初级：今天的小步迭代
选你今天要做的任何一件事（写代码、写文档、做饭都可以）：
1. 先做最小、最简的版本——"能用就行"的第一版，限制在 15 分钟内完成
2. 做完后，写下 2 个"下一版可以改进的地方"
3. 再做第二版——只改这 2 个地方，限制在 10 分钟内
4. 比较两版的差异：比直线做到完美少用了多少时间？结果你满意吗？

### 中级：回顾一个延期项目
找一个你曾经"想一次做好但一直没开始"的事情：
1. 写出如果只做第一版会是什么样子（2-3 个核心功能）
2. 计算如果每周花 2 小时迭代，2 个月后你能迭代几次
3. 假设每次迭代改进 15%，2 个月后的版本比第一版好了多少？
4. 这和"等完美的第一版"哪个更可能真正完成？

### 高级：设计一个迭代系统
为你的工作/生活设计一个长期迭代机制：
1. 选择一个你想持续改进的领域（技能、健康、知识、关系）
2. 定义你的"最小可交付版本"——什么是你现在就能做的？
3. 设定迭代节奏——每周一次还是每天一次？
4. 定义"回顾"环节——每次迭代结束时，回答三个问题：
   - 这次比上次好了多少？
   - 最大的意外发现是什么？
   - 下次最优先改进什么？
5. 运行 3 个迭代周期后，复盘：这个机制是否可持续？

### 反思
- 你上次说"等我想清楚了再开始"是在什么时候？如果当时用迭代思维，你会怎么做？
- 迭代的隐形前提是"接受不完美"。你目前最需要接受哪个方面的"暂时不完美"？
