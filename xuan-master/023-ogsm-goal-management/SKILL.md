# 🎯 德鲁克目标管理（OGSM）— 模型 023

> **版本**: 1.0.0 | **作者**: Arthur

## 核心定义

将战略意图逐层转化为可执行行动的框架——从 **Objective**（目的/方向）出发，到 **Goals**（具体目标/量化指标），到 **Strategy**（策略路径/实现手段），再到 **Measure**（测量标准/进度检查）。根本洞察：**战略不是「想做什么」，而是「四个层次一以贯之的因果关系链」。**

## 四层结构

### O — Objective（目的）
定性、长期、方向性的"为什么"——灵魂。
- 回答：我们为什么存在？我们为什么要做这件事？
- 特点：一句有力的话，不需要量化，但必须是清晰的
- **错误的 O**："我们要成为最好的企业"（太模糊）
- **好的 O**："让东南亚在 3 年内成为我们增长最快的区域市场"

### G — Goals（目标）
定量、短期、可衡量的"是什么"——里程碑。
- 回答：到什么时间达到什么数字？
- 特点：SMART（具体、可衡量、可达、相关、有时限）
- **注意**：Goals 是 Outcome（结果目标）而非 Output（产出目标）

### S — Strategies（策略）
定性的路径选择"怎么做"——杠杆点。
- 回答：我们怎么实现这个目标？我们选择什么路径？
- 特点：每个 S 是一个**战略选择**（不是待办事项）
- **关键**：S 必须具体到可以"否定其他选项"的程度

### M — Measures（度量）
定量的跟踪指标"怎么知道"——仪表盘。
- 回答：我怎么知道策略在奏效？我应该跟踪哪些具体指标？
- 特点：每个 S 应有 1-3 个 M，提供"策略是否有效"的实时信号
- **O 与 G 的 M**：季度检查（OG 层的 M）
- **S 的 M**：月/周跟踪（执行层的 M）

## 覆盖领域（8 个）

| # | 领域 | OGSM 价值 |
|---|------|-----------|
| 1 | **企业战略执行** | 核心应用场景——从 CEO 战略意图到部门执行 |
| 2 | **OKR / KPI 体系设计** | OKR（O+KRs）≈ OGSM 的 O+G；M 和 KPIs 的区别 |
| 3 | **产品路线图规划** | O=产品愿景，G=季度指标，S=特性路径选择 |
| 4 | **项目管理** | O=项目目标，G=交付标准，S=实施路径 |
| 5 | **个人年度/季度规划** | O=人生愿景，G=年度目标，S=策略/习惯，M=每日跟踪 |
| 6 | **团队目标对齐** | 每个团队的 OGSM 必须对齐上级组织的 O |
| 7 | **非营利/公共项目管理** | 不以利润为导向同样适用——O 可以是非财务性的 |
| 8 | **创业公司从 0 到 1** | 战略锚定——O 就是创业的"为什么" |

## 核心原则（7 条）

### 1. 从 O 开始
所有层级从 O 向下"流"——没有清晰的 O，G 可能偏离方向；没有明确的 G，S 没有目标对标。

### 2. 因果关系链
**O→G→S→M 是因果链，不是愿望清单。** 每一层要能回答"为什么上一层能通过这一层来实现"。

### 3. 对齐（Alignment）
O 只有一个。你的 O 必须是对上级组织 O 的承接（即你的 O = 为了实现上级的 G 或 S）。

### 4. 5S 法则
每层不超过 5 个条目。O=1个，G=1-3个，S=1-5个，M=每个 S 对应 1-3 个。

### 5. 两种度量
- G 的 M = **结果指标**（Result Indicator）——终点对不对
- S 的 M = **过程指标**（Progress Indicator）——路在不在走

### 6. 季末检验因果
"这个季度末我的 M 过了吗？→ 如果没过，是我的 S 错了还是执行出了问题？"

### 7. 一个人的 OGSM 同样成立
个人规划完全可以用 OGSM：O=人生方向，G=年度目标，S=核心习惯，M=每日打卡。

## 实践要点

### 一页纸 OGSM 格式
```
┌─────────────────────────────────────────────┐
│             O（一句话目的）                    │
├──────────────┬──────────────────────────────┤
│ G（目标）     │ S（策略）                     │
│ 1.           │ 1.                           │
│ 2.           │ 2.                           │
│ 3.           │ 3.                           │
├──────────────┴──────────────────────────────┤
│ M（度量）                                    │
│ G1-M: ________   S1-M: _________            │
│ G2-M: ________   S2-M: _________            │
│ G3-M: ________   S3-M: _________            │
└─────────────────────────────────────────────┘
```

### ⚠️ 反模式

| 反模式 | 表现 | 解药 |
|--------|------|------|
| O 太大 | "成为全球领先的 X"→下面执行层完全不知道O如何落地 | O 必须能具体到可以被 G 证明或证伪 |
| G 和 M 混为一谈 | "我的 G=用户增长10%"→M="每个月看用户数"→因果链断裂 | G是"去哪儿"，M是"到了没"，两者不可互换 |
| S 是待办事项不是策略 | "招更多销售"是待办，"建立销售激励体系"是策略 | S 是决策，不是行动 |
| 一年写一次就锁死 | OGSM 写完就放在抽屉里直到年底 | OGSM 需季度回顾、月度 M 检查 |

## 典型案例

### 1. 某 SaaS 企业年度转型 OGSM
- **O**：从"工具型产品公司"转型为"平台型生态公司"
- **G**：
  1. API 调用量 ×10 倍
  2. 第三方开发者达到 1000 人
  3. 平台收入占比从 10% → 30%
- **S**：
  1. 发布公共 API 网关+开发者文档
  2. 建立开发者社区+奖励计划
  3. 与 3 个行业标杆 ISV 建立合作
- **M**：
  - API 调用量周趋势
  - 开发者注册数
  - ISV 合作进展
- **结果**：年度 API 调用量 ×8 倍，平台收入占比 25%——接近但未完全达到目标 S 策略基本正确，但 S2（社区运营）投入相对于 S1 少了 30%。

### 2. OKR vs OGSM 对比
- **OKR 的问题**：Objective 有时与大量 Key Results 放在一起导致决策优先级混乱
- **OGSM 的优势**：将 Goals（去哪）与 Measures（怎么知道到了没）明确分离，且强化了 S（怎么去）的思考和选择

### 3. 个人季度 OGSM
- **O**：成为全栈开发工程师、能独立交付一个完整的 SaaS 项目
- **G**：在 Q2 结束前完成一个带认证+支付功能的 MVP
- **S**：
  1. 每天 1 小时专注学习 + 2 小时项目编码（替代碎片化学习）
  2. 每两周发布一个可展示的版本（倒逼写完整代码）
  3. 找一个 peer 做每周 code review（引入外部反馈）
- **M**：
  - 连续学习天数和编码时长
  - 功能完成度 checklist
  - code review 中被指出的技术债务数

### 4. 产品路线图的 OGSM 分解
- **O**：提高用户留存率（核心痛点）
- **G**：Q3 末 30 天留存从 40% 提升到 60%
- **S**：
  1. 优化新用户引导流程
  2. 引入"每周推荐内容"的个性化推送
  3. 建立用户"失败时刻"的主动跟进机制
- **M**：
  - 第 1 周引导完成率（S1 的 M）
  - 推送打开率与功能转化率（S2 的 M）
  - 主动跟进后的用户七日折返率（S3 的 M）

### 5. 从使命到行动的连续链
- **使命**："提升全球教育的可及性"（太宽——需要 OGSM 来拉近）
- **O**（5 年）：让印度农村 50 万学生在三年内通过数字化课程获得高中教育
- **G**（年度）：第一年覆盖 10 万学生、课程完成率 ≥70%
- **S**（Q1）：与 5 个本地 NGO 建立线下学习中心网络
- **M**：每个学习中心的运营数据周报

- **OGSM(023) + 分层(001)**：OGSM 四层本身就是分层思想的具体应用：O（愿景层）→ G（目标层）→ S（策略层）→ M（执行层）。
- **OGSM(023) + SWOT(018)**：SWOT 告诉你"外部环境+内部能力"的全貌，OGSM 告诉你在 SWOT 基础上"接下来怎么做"—SWOT 是诊断，OGSM 是处方。
- **OGSM(023) + 反馈循环(011)**：OGSM 的季末检讨本质上是一个反馈循环节点：M vs G 的偏差 → 修正 S → 重新验证 O→G 的因果关系。

- **OGSM(023) + 5W2H(014)**：OGSM 的每个层次都可以用 5W2H 做深度追问来提升质量（特别是 O 和 S 层）。

## 💡 实战练习

### 初级
初级：为你当前最重要的目标写一个 OGSM：目的（为什么）、目标（衡量什么）、策略（怎么做到）、度量（怎么知道在做）。确保"策略"和"目标"之间有可验证的逻辑链。

### 中级
季度复盘：回顾上季度的OGSM——目标达成了多少？策略有效吗？度量准确吗？最关键的：你在什么假设上犯了错误？下季度如何修正？

### 高级
为你的团队或项目建立OGSM季度循环。关键不是"写出OGSM文档"，而是"每个季度诚实地面对度量数据"。设计你的第一次诚实地回顾。

### 反思
OGSM最有价值的不是"G"而是"S"——策略回答的是"为什么我们选择这样做而不是那样做"。你上次的策略选择，有替代方案吗？为什么排除了它们？
